# AGENTS.md

## Назначение проекта

Этот репозиторий предназначен для локальной Russian-first RAG-системы.
Цель системы:
- работать полностью локально;
- маршрутизировать запросы между `memory`, `rag` и `direct`;
- использовать отдельные слои `LLM`, `embeddings`, `retrieval`, `rerank`, `memory`;
- минимизировать задержку и расход памяти;
- отвечать преимущественно на русском языке.

Базовая архитектура описана в [rus_rag_local_architecture.md](D:\repos\RAG\rus_rag_local_architecture.md).

## Рабочие роли

### 1. AI Engineer
Используется для:
- реализации RAG-пайплайна;
- интеграции локальных моделей;
- подготовки чанков, эмбеддингов, rerank-логики;
- оценки качества retrieval/rerank;
- оптимизации inference и памяти.

Ожидания:
- проектировать решение под локальный запуск;
- не тащить облачные зависимости без явной причины;
- учитывать RAM-лимиты и стоимость каждого шага пайплайна;
- держать интерфейсы между моделями простыми и заменяемыми.

### 2. Backend Architect
Используется для:
- проектирования модулей, API и сервисных границ;
- схем данных, индексов, хранилищ и файловой организации;
- устойчивости, логирования, конфигурации и расширяемости системы.

Ожидания:
- строить решение как набор простых, слабо связанных компонентов;
- явно разделять router, retrieval, rerank, memory и generation;
- избегать скрытой магии и сильного связывания модулей;
- проектировать код так, чтобы модели можно было заменить без переписывания всей системы.

### 3. Code Reviewer
Используется для:
- проверки корректности маршрутизации;
- поиска лишних RAG-вызовов;
- проверки обработки пустого/слабого контекста;
- проверки тестов, отказоустойчивости и регрессий.

Фокус ревью:
- корректность;
- надежность;
- читаемость;
- производительность;
- покрытие ключевых сценариев.

### 4. Trend Researcher
Для этого репозитория роль вторична.
Используется только когда нужно:
- сравнить локальные модели, ранкеры, векторные БД;
- оценить внешние подходы перед архитектурным решением;
- собрать актуальные практики по локальному RAG.

По умолчанию не должна влиять на основной кодовый поток.

## Правила проектирования

### Router-first
Любая новая логика должна уважать главный принцип проекта:
- не каждый запрос требует RAG;
- сначала выбирается маршрут;
- потом собирается только нужный контекст.

Допустимые маршруты:
- `memory`
- `rag`
- `direct`

Если маршрут неясен, код должен выбирать наиболее дешевый и безопасный путь, а не автоматически запускать полный RAG.

### RAG отдельно от памяти
Нельзя смешивать chat memory и knowledge context без явной причины.
Нужно:
- отдельно получать историю диалога;
- отдельно получать документы;
- явно контролировать, что попадает в итоговый prompt.

### Retrieval с высоким recall, rerank с высоким precision
При изменениях учитывать:
- retrieval возвращает кандидатов;
- rerank отсекает шум;
- в LLM уходит только короткий релевантный контекст.

Не допускается:
- передавать слишком много чанков;
- пропускать rerank без причины;
- собирать контекст "на всякий случай".

### Локальность и экономия ресурсов
Система должна оставаться пригодной для машины с ограниченной памятью.
Поэтому приоритет:
- простые зависимости;
- переиспользуемые модели;
- короткий контекст;
- минимальное число вызовов моделей;
- отсутствие сетевой зависимости в основном сценарии.

## Ожидания к реализации

### Python
- Для локального запуска и установки модулей всегда использовать `.venv`.
- Если добавляется Python-код, команды должны быть совместимы с `.venv`.
- Не предполагать глобально установленные пакеты.

### Структура кода
Предпочтительно разделять код по зонам ответственности:
- router;
- embeddings;
- retrieval;
- rerank;
- memory;
- prompt/context assembly;
- generation/inference;
- config;
- evaluation/tests.

### Конфигурация
Все меняемые параметры должны быть вынесены в конфиг:
- имена моделей;
- `top_k`;
- число чанков в финальном контексте;
- размеры чанков и overlap;
- пороги router/rerank;
- лимиты истории.

Не хардкодить такие значения глубоко внутри бизнес-логики.

### Отказоустойчивость
Нужно предусматривать:
- пустую выдачу retrieval;
- слабые или нерелевантные результаты rerank;
- недоступность локальной модели;
- превышение лимита контекста;
- fallback в `direct`, когда это безопасно.

## Ожидания к тестам

Минимально важные тесты:
- router корректно различает `memory`, `rag`, `direct`;
- RAG не вызывается для простого диалога;
- при `rag` контекст ограничивается заданным числом чанков;
- при отсутствии хороших документов система не ломается;
- сборка prompt не смешивает нерелевантные источники;
- ключевые параметры читаются из конфига.

Если меняется логика маршрутизации или контекстной сборки, тесты обязательны.

## Стиль решений

Нужно:
- писать просто и явно;
- держать интерфейсы короткими;
- объяснять нетривиальные решения короткими комментариями;
- выбирать предсказуемое поведение вместо "умной" неявной логики.

Не нужно:
- добавлять абстракции без реальной необходимости;
- усложнять пайплайн ради будущих гипотез;
- подключать облачные сервисы в базовую архитектуру;
- строить систему вокруг LLM, игнорируя router/retrieval/rerank.

## Критерий качества изменений

Хорошее изменение в этом репозитории:
- усиливает локальный RAG pipeline;
- уменьшает лишние вызовы;
- делает маршрутизацию понятнее;
- ограничивает контекст и шум;
- упрощает замену моделей;
- не ломает русский-first сценарий.
